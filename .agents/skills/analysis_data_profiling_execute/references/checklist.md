# Step 3: データ収集・プロファイリング（Data） 実行チェックリスト

分析プロジェクトにおける「Step 3: データの健康診断（プロファイリング）」を自律的に実行する際、以下のベストプラクティスとチェックポイントを満たすようにコードを記述・実行してください。
高度な前処理（特徴量作成等）はここでは行わず、Step 4へ渡すための「安全確認」と「構造理解」に徹してください。

## 1. 抽出条件と安全性の確認
- **スキャン量チェック（必須）**: SQLを実行する前に、必ずDry-RunやEXPLAINでコスト・データ量を見積もり、閾値（1GB等）を超える場合はユーザーに承認を求めているか。
- **【最重要】PII（個人情報）の除外・マスキング**: 氏名、メールアドレス、電話番号などの個人特定情報（PII）を無防備に抽出していないか。分析に不要な場合はカラムをDROPし、必要な場合（ユーザー追跡等）は必ずハッシュ化等のマスキング処理を行うこと。
- **Fan-outの防止**: テーブル結合（JOIN）時に、意図せずレコード数が倍増（Fan-out）していないか、一意性（ユニークキー）を確認しているか。

## 2. 構造理解とビジネスルールの推測
- **ER図・テーブルリレーションの把握**: 対象となる複数のテーブルが、どのようなキーで紐付き、1対多などのどのような制約を持っているかを把握しているか。
- **ビジネスルールの推測**: カラムの値の分布から、「このフラグが1の場合は〇〇というビジネス状態を意味している」などの特性を言語化・推測しているか。

## 3. 基礎プロファイリング（データ品質チェック）
- **欠損値（NULL）**: 各カラムの欠損割合を算出し、分析に与える影響を評価しているか。
- **異常値・外れ値**: 年齢がマイナス、購入金額が異常に大きいなど、ビジネスロジック的にあり得ない値を検知しているか。
- **分布の確認**: 主要な変数の分布（ヒストグラム）やカーディナリティ（カテゴリの種類数）を確認し、分析に十分なサンプルサイズがあるかを評価しているか。

## 4. Step 4への申し送り事項・アラートの作成
- データ品質の確認結果を踏まえ、次にStep 4を実行するアナリスト（またはエージェント）に対し、「このデータを使う上での注意点、限界、リスク」を明確なアラートとして申し送っているか。
